## 一、执行摘要

**TodoAgent**（[todoagent.cc](https://todoagent.cc)）是一款面向「深度工作与 AI 协作」的 SaaS：以待办为轴心，把目标、约束、验收标准与对话上下文沉淀在同一工作流里，并辅以「任务可替代性分析」、可复用 SOP 记忆与可追踪的执行日志，帮助知识工作者在可控前提下逐步把人机分工做实。

**商业逻辑**：采用 **Freemium + 订阅**——免费用户每日（UTC）享有有限次数的 Agent 对话，付费订阅解除额度并覆盖模型与基础设施成本。**技术栈**为 React / Vite、Clerk 身份、Supabase 数据、Moonshot（Kimi）推理、Vercel 托管与 **Stripe Checkout 订阅 + Webhook 权益闭环**。

**阶段判断**：产品已完成核心闭环与计费基础设施接入，处于 **PMF 验证与增长实验** 前期；法律文本已预留订阅条款路径，上线收费前需完成协议/隐私与订单页的同步更新。

## 三、市场机会与痛点

### 3.1 趋势

- 通用大模型普及后，用户痛点从「会不会用 AI」转向「如何把 AI 嵌进稳定工作系统」。
- 对话类产品上下文易失、难以与「下一步动作」绑定；传统待办缺少对「哪些交给 AI、哪些必须人做」的结构化判断。

### 3.2 目标客户痛点

- **上下文碎片化**：决策散落在 IM、PR 评论、文档与模型窗口之间，周复盘难以还原。
- **任务定义模糊**：标题只有动词、缺少验收与约束，Agent 反复追问或误执行。
- **自动化不可控**：希望减负，但担心黑箱；需要可审核、可回退的渐进自动化路径。

TodoAgent 将 **任务定义、对话、分析与 SOP** 收敛到同一轴心，直接回应上述摩擦（与落地页、PSEO 人设文案一致）。

## 四、产品与服务

### 4.1 核心能力（与实现一致）

1. **任务聚焦模式**：目标、约束、完成标准与材料挂载在单条待办上。  
2. **可替代性引擎**：对已完成任务做结构化拆解，输出「可替代 / 不可替代 / 不清楚」等结论，支撑人机分工决策。  
3. **AI 执行交接**：在确认前提下推进可替代步骤（具体边界以实际上线功能为准）。  
4. **SOP 记忆**：完成闭环后沉淀可复用手册式内容。  
5. **协作闭环**：执行—反馈—复盘—复利；支持效率看板类指标（以产品实际展示为准）。  
6. **计费与权益**：服务端代理模型调用；免费日配额 + Stripe 订阅判定（见 `i18n` 与 API 实现）。

### 4.2 路线图（建议）

| 阶段 | 重点 |
|------|------|
| 0–3 个月 | 转化漏斗：注册→首日激活（首条高质量任务）→次日回访；付费页与订阅稳定性 |
| 3–6 个月 | 垂直场景模板（工程师 / PM / 分析师等，与现有 PSEO 人设联动） |
| 6–12 个月 | 团队版（若需求验证）：共享空间、权限、用量与账单（需单独架构与合规评估） |

## 五、商业模式与定价

### 5.1 收入结构

- **主要**：B2C 订阅（Stripe 周期性扣款）。  
- **潜在扩展**：B2B 团队席位、更高 SLA、专属模型或企业部署（长期选项，需单独商业与合规路径）。

### 5.2 定价原则

- **单位经济**：订阅价需覆盖 **LLM Token、存储、身份、支付手续费、客服与合规** 边际成本，并保留毛利。  
- **心理锚点**：免费档控制滥用与成本；付费档强调「稳定系统 + 不限对话（或更高配额）」而非单纯「多一个聊天窗口」。  
- **地区**：Stripe 支持币种与税负因地区而异，需在结账页与发票策略中写清。

### 5.3 当前产品中的计费事实（便于与法务对齐）

- 界面文案体现：**未订阅用户每自然日（UTC）10 次免费 Agent 对话；订阅后可继续使用**（具体以线上版本与订单页为准）。  
- 用户协议当前表述为**暂不收费、未来可能订阅**——**正式收费前必须更新协议/隐私与结账条款**，避免披露与产品不一致。

## 六、竞争与定位

| 类型 | 代表方向 | TodoAgent 的站位 |
|------|-----------|------------------|
| 传统待办 | Things、Todoist、滴答清单等 | 强化 **任务级 AI 与可替代性分析**，而非只做提醒与列表 |
| 通用 AI 聊天 | ChatGPT、Kimi 网页等 | 强化 **待办结构、可追溯日志与 SOP 复利** |
| AI Agent 通用平台 | 各类「自主 Agent」 | 强调 **人在环、渐进自动化**，降低信任与合规顾虑 |

定位一句话：**「给已经每天用 LLM 的规划者/创造者用的、可追踪的待办操作系统。」**

## 七、获客与增长（Go-to-Market）

1. **内容与人设 SEO**：已有多角色 PSEO 素材（工程师、PM、分析师等），可持续扩展长尾词与案例页。  
2. **社区与开发者关系**：开源/博客/方法论页（「四天工作制」「注意力工作流」）与产品叙事一致时转化更高。  
3. **产品驱动增长**：免费配额足够完成「啊哈时刻」（例如完成首次可替代性分析或第一条闭环复盘），再触发付费墙。  
4. **推荐与裂变（可选）**：邀请奖励需评估成本与羊毛风险，建议订阅稳定后再做。

## 八、运营与技术

- **可靠性**：Webhook + `confirm-checkout-session` 兜底（仓库已有实践总结），避免「已付款未开通」。  
- **安全**：Clerk JWT 校验、Stripe Session 与用户绑定校验、Supabase RLS（按你方实际策略执行）。  
- **观测**：订阅转化、日活、Agent 次数分布、模型错误率、Checkout 失败原因。  
- **依赖风险**：第三方 LLM 涨价/限流、身份或支付服务区域政策变化——需保留备用模型或降级策略的架构空间。

## 九、合规与风险摘要

- **个人信息与跨境**：隐私政策需与实际数据处理（Clerk、Supabase、Moonshot、Stripe、托管方）一致；涉及跨境传输时按司法辖区要求更新条款与用户同意。  
- **支付与消费者保护**：自动续费、退款、发票规则在收费前须在订单页与协议中明确。  
- **AI 内容**：输出仅供参考、不构成专业建议；高风险场景（医疗、法律、投资）需在界面与协议中免责提示。  
- **主体形态**：自然人运营适合早期验证；规模化后评估公司化、税务与知识产权归属。
